# 热门龙头 / 共识情报 — 合并方案（实现前设计稿）

本文合并以下诉求：

- 龙头定义：**不仅是涨停股**，而是**板块（或题材）上涨的领头羊**；分析目标包含「**为何是龙头**」。
- **集合竞价时段**定时拉起分析，结合**历史信息**，判断**当日可能龙头**。
- 与既有约定合并：**轮询与 AB 对齐 45s**、**已推送股票无更新不反复推**、**新闻足够重要且可能影响股市再推**、**微信超长分段发送**、**与 ab-stock-quant 进程/数据隔离**。

以下均为**方案层**描述，不含代码实现细节。

---

## 一、龙头定义（扩展）

| 概念 | 含义 |
|------|------|
| **狭义（情绪）** | 连板高标、涨停池前排（当前实现偏此）。 |
| **广义（板块领头羊）** | 在**同一板块或题材**内，**涨幅/成交额/带动性**居前、对板块指数或跟风票有**示范效应**的标的；**未必涨停**（如大票中阳、小票领涨未封板）。 |

**分析输出应回答**：该标的相对**谁**领先（板块指数？题材成分？）、**领先的表现形式**（涨幅、换手、封单、人气跃迁）、**叙事与资金是否同向**——即「**为什么是龙头**」，而非只列代码。

---

## 二、分析策略：如何识别并解释「龙头」

建议采用 **多源 + 分层**，与现有「融合榜 + 财联社整合」衔接并扩展：

1. **板块/题材锚定**  
   - 来源：行业、概念板块涨幅榜、成分股涨幅排序（东财等公开接口）。  
   - 在板块内找 **相对强度** 前列（涨幅、换手、成交额占比），标记为 **候选领头羊**。

2. **与涨停池/人气榜交叉**  
   - **交集**：既是板块内前排，又在涨停/飙升/人气中 → **强龙头候选**。  
   - **仅板块强、未涨停**：单独标签 **「趋势/中军龙头」**，避免被「必须涨停」过滤掉。

3. **「为何是龙头」的可解释字段（方案层）**  
   - **板块内名次**、**相对板块指数超额**（若可算）。  
   - **叙事**：财联社/整合分析中与该板块一致的描述（**不逐条列新闻**，见前文门控）。  
   - **历史类比**：见下一节「历史信息」。

4. **DeepSeek 的角色**  
   - 在**规则层特征**已给出的前提下，做 **归纳与因果表述**（禁止编造数值）；输出 **「龙头类型」**（情绪高标 / 中军 / 补涨跟风等）与 **一句话逻辑**。

---

## 三、集合竞价时段：定时拉起 + 早盘判断

### 3.1 目标

在 **集合竞价及竞价结束后的关键节点**，**定时触发**分析任务（可与盘中 `monitor.py --loop` **不同入口或同进程不同阶段**），结合 **历史与隔夜信息**，输出 **「今日可能龙头方向与候选」**，避免与盘中每分钟推送混成一团。

### 3.2 时间点（上海时间，交易日）

| 时段 | 说明（方案） |
|------|----------------|
| **9:15–9:20** | 可尝试拉取**竞价阶段**可见的行情/涨跌幅快照（以接口能力为准）；若数据不可用则**跳过本段仅记日志**。 |
| **9:20–9:25** | 同上；关注**价量意愿**变化（若接口提供）。 |
| **9:25 前后** | **竞价价格基本确定**，优先做一次 **「竞价结果快照」** 分析。 |
| **9:30 开盘后不久** | **开盘确认**后再跑一次（例如 9:31 或 9:35），用**真实连续竞价**数据修正早盘判断。 |

具体 cron/systemd **分钟级**配置在实现阶段写死为可配项（如 `HC_AUCTION_CRON` 或独立 `auction_run.py`）。

### 3.3 早盘分析内容（与盘中区分）

- **输入**：隔夜财联社高等级摘要（若有）、**昨日**涨停池/龙虎榜/板块强弱、**当日竞价**快照（若有）、**历史统计**（见下）。  
- **输出**：**方向（板块/题材）+ 候选票 + 依据类型**（竞价强势、昨日延续、叙事匹配等）；**不保证**与盘中完全一致，标注 **「早盘预案」**。  
- **推送**：受 **「新闻/早盘专项」冷却与条数上限** 约束，避免与 45s 盘中推送叠加爆炸（见第六节）。

---

## 四、历史信息：用来判断「今日可能龙头」

实现时按需拉取（**只读公开数据**），与 AB 池文件**隔离**：

- **昨日**：涨停池延续性、龙虎榜净买/席位类型（盘后）、板块涨幅排名。  
- **近 N 日**：板块轮动、某题材是否**反复活跃**（频次、涨幅分位）。  
- **可选**：与「昨日整合结论」对比，看 **主线是否延续或切换**。

用途：

- **集合竞价/早盘**：判断 **资金可能延续昨日主线还是高低切**。  
- **盘中**：仅当 **龙头特征相对历史快照发生变化** 时再推（与「无更新不反复推」一致）。

---

## 五、与前一版「防信息大爆炸」方案合并

| 维度 | 策略（合并后） |
|------|----------------|
| **轮询间隔** | 与 AB 统一 **45s**（`HC_POLL_INTERVAL_SEC=45`）。 |
| **股票推送** | 维护**每票/每榜快照**；仅 **进榜、出榜、连板/封板质量/板块角色质变** 等算更新；避免同一票无变化反复刷屏。 |
| **新闻/财联社** | **等级 + 关键词 + 时间窗合并**；可选 **模型门控**「是否可能影响股市」；**digest 与突发**分级；**每小时条数上限**。 |
| **微信** | **超长自动分段**（已实现方向）；早盘/盘中可共用同一 webhook 但 **标题区分**。 |
| **龙头定义** | 分析层显式包含 **板块领头羊** 与 **涨停高标** 两类，并写清 **为何是龙头**。 |

## 六、推送节奏总控（避免叠加）

建议 **分流**：

| 类型 | 典型频率（方案） |
|------|------------------|
| 盘中榜单 **增量** | 依赖「有更新」判定 + 冷却。 |
| 财联社 **整合 / 高影响** | 长间隔 digest + 少量突发。 |
| **集合竞价 / 早盘** | 每日固定 **2～4 次**（见第三节），单独计数冷却。 |

同一自然日内 **总推送条数**可设硬上限（实现时配置）。

---

## 七、与 ab-stock-quant 的边界（重申）

- **进程与仓库**：独立；不读写 `target_pool_ab_shape.csv` 等。  
- **可选**：仅当用户配置时，**交叉比对**「情报候选」与「自选池」用于加粗提示（默认关闭）。

---

## 八、实现顺序建议（供下一版开发拆解）

1. 配置默认 **45s**；状态结构支持 **股票快照 diff** 与 **新闻门控**（规则优先）。  
2. 扩展数据源与融合逻辑：板块内 **领头羊** 与 **涨停高标** 双轨标签 + 「为何龙头」字段模板。  
3. **集合竞价/早盘** 独立调度与输出模板；挂接 **历史序列**。  
4. 再收紧 **推送合并** 与 **DeepSeek 调用频次**。
